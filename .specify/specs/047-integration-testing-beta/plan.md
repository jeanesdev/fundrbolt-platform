# Implementation Plan: Beta-Readiness Integration Test Suite

**Branch**: `047-integration-testing-beta` | **Date**: 2026-05-30 | **Spec**: `.specify/specs/047-integration-testing-beta/spec.md`
**Input**: Feature specification from `/specs/047-integration-testing-beta/spec.md`

## Summary

Build a comprehensive, automated integration test suite covering every critical donor, admin, auctioneer, and staff flow before opening FundrBolt to beta users. The suite consists of: (1) a fast PR-gate Playwright + pytest suite that blocks merges on critical regressions, (2) a nightly comprehensive Playwright suite against staging with Teams + GitHub Issue notifications, (3) an idempotent seed system for deterministic test environments, and (4) a manual pre-beta checklist for real-device and real-payment-processor validation.

Key technical approach: Mailpit for email capture, pytest-freezegun + Playwright clock API for time simulation, existing stub payment gateway, top-level `e2e/` Playwright project for cross-PWA flows, and two new GitHub Actions workflows.

## Technical Context

**Language/Version**: Python 3.11+ (backend API tests / seed), TypeScript 5.x (Playwright E2E tests)
**Primary Dependencies**: pytest 7.4+, pytest-asyncio 0.21+, httpx 0.25+, freezegun 1.5+, pytest-freezegun 0.4+, @playwright/test ^1.58.2
**Storage**: PostgreSQL 15 (seed + test DB), Redis 7 (existing) — no new schema changes
**Testing**: This feature IS the testing infrastructure; all CI commands are part of the deliverable
**Target Platform**: Linux (GitHub Actions ubuntu-latest), Docker Desktop (local dev), Playwright Chromium + WebKit
**Project Type**: Web application (monorepo — backend + two frontend PWAs)
**Performance Goals**: PR gate ≤ 8 minutes end-to-end including env bringup (SC-001); seed idempotency verified in < 30 s
**Constraints**: No new production DB tables; no test-only API routes; no real payment gateway; no real email delivery; stub gateway only; Mailpit only
**Scale/Scope**: 53 functional requirements, 4 user stories, ~7 critical-path flows (PR gate), ~47 full-suite flows (nightly)

## Architecture Phases

### Phase 1: Infrastructure (Blocking Prerequisites)
- Mailpit service added to `docker-compose.yml`
- `docker-compose.test.yml` override for test-mode email backend
- `tests/seed/` directory and seed script skeleton
- `e2e/` Playwright project initialized with `package.json` and configs
- `Makefile` targets: `seed`, `test-e2e`, `test-critical-path`

### Phase 2: Seed System (US3)
- Idempotent seed covering all entities required by FR-050
- Per-entity fixture modules with `ON CONFLICT DO UPDATE` or `DO NOTHING`
- `--tenant-slug` flag for staging automation-tenant scoping
- Python provisioning helpers (`tests/seed/helpers.py`) for mutation scenarios

### Phase 3: PR Gate — Critical Path (US1)
- 7 Playwright critical-path spec files
- `critical-path-gate.yml` GitHub Actions workflow with sharding
- Mailpit helpers (`e2e/helpers/email.ts`) for email assertion
- Time helpers (`e2e/helpers/time.ts`) using `page.clock`
- Base fixtures (`e2e/fixtures/base-fixtures.ts`)

### Phase 4: Extended API Integration Tests
- New pytest files for FRs not yet covered in existing `backend/app/tests/integration/`
- Uses existing conftest, db_session, async_client fixtures
- `freezegun` for time-sensitive scenarios

### Phase 5: Nightly Full Suite (US2)
- All `e2e/full-suite/**/*.spec.ts` files
- `nightly-regression.yml` workflow with matrix over spec groups
- Teams webhook notification step (FR-045b)
- GitHub Issues create/update/close step (FR-045c)

### Phase 6: Manual Checklist (US4)
- `tests/manual/pre-beta-checklist.md` covering all out-of-automation items (FR-052)
- Includes structured evidence format for each item

## Complexity Tracking

| Complexity | Why Needed | Simpler Alternative Rejected Because |
|------------|------------|--------------------------------------|
| Two separate CI workflows | PR gate and nightly have incompatible trigger, timing, and notification requirements | Conditional logic in one workflow would be unreadable and harder to disable independently |
| Top-level `e2e/` Playwright project | Cross-PWA flows span admin and donor URLs simultaneously | Single-PWA Playwright projects cannot orchestrate both apps without shared config duplication |
